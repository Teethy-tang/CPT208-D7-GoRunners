Usability Testing

Alpha prototype testing, results, findings, and supporting evaluation data.

5.1 Usability Testing with Alpha Prototype

Purpose

To evaluate whether MoodRun's Alpha prototype was understandable, usable, and emotionally supportive, we conducted a small-scale usability test with three target users. The test focused on whether users could complete the core journey smoothly and whether the emotion-aware interaction helped reduce the pressure of starting a run.

Participants

Participant Profile Reason for Selection
P1 University student, occasional runner Represents young adults with study pressure and irregular exercise habits.
P2 Young adult with interest in fitness apps Familiar with running apps and able to compare MoodRun with metric-driven tools.
P3 University student who rarely runs Represents users who may face psychological barriers when starting exercise.

Testing Method

Each participant was asked to interact with the Alpha version of MoodRun and complete the following tasks:

  1. Start a run from the home page.
  2. Complete the mood check-in by selecting a mood and entering a short reflection.
  3. Choose or accept a recommended running plan.
  4. Enter the running screen and understand the displayed run data.
  5. Complete the run summary and use the Wisdom Book reflection feature.
  6. Visit the profile page to review previous emotional and running records.

During the test, we observed whether users could complete each task without guidance, noted hesitation points, and collected short verbal feedback after the session.

5.3 Key Findings

Main insights from observation and participant feedback.

Finding 1: Mood check-in made the app feel more personal

Participants understood the emotional purpose of the system quickly. They felt that starting with a mood selection made the experience feel less intimidating than opening a traditional fitness tracker.

"The mood check-in made starting easier because it asked how I felt first."

Finding 2: Users liked the supportive tone, but wanted clearer recommendation logic

Most participants liked the idea of receiving a mood-based running plan. However, one participant wanted the system to explain why a certain plan was suggested.

Design implication: add a short explanation under the recommended plan, such as "Recommended because you selected 'stressed': a gentle jog can help release tension without pressure."

Finding 3: The running screen was useful, but should avoid feeling too performance-heavy

Participants appreciated seeing distance, pace, time, and progress. However, one user commented that too many numbers could make the app feel similar to normal running apps.

Design implication: keep key running data visible, but visually balance it with supportive feedback, voice encouragement, and mood-based messages.

Finding 4: Reflection was the most emotionally meaningful part

All three participants responded positively to the Wisdom Book feature because it connected the user's pre-run thought with a post-run emotional message.

"The reflection part helped me notice how I changed after running."

Iterative Refinement Based on Finding 2

Clearer recommendation logic in the Choose Your Run screen.

Finding 2: Users liked the supportive tone, but wanted clearer recommendation logic

During the Alpha usability testing, users responded positively to MoodRun's supportive and emotion-aware tone. However, one participant pointed out that the mood-based run recommendation was not fully transparent. They understood that the app suggested a plan, but wanted to know why that plan matched their selected mood.

Before

In the Alpha prototype, the Choose Your Run screen displayed a recommended plan with basic running information, such as plan name, distance, duration, and intensity. However, it did not clearly explain how the selected mood influenced the recommendation. As a result, the suggested plan could appear like a generic fitness preset rather than an emotionally adaptive decision.

After

Based on this finding, we improved the Choose Your Run screen by adding a visible recommendation explanation under the recommended plan. The updated card now includes a "Why This Fits" section, which shows the selected mood and explains why the suggested distance, pace, and intensity match that emotional state.

Example Recommendation Explanation

Recommended because you selected STRESSED:
This focused high-energy pace helps release pressure without asking you to sit still with it.

This change made the recommendation logic clearer and more emotionally meaningful. Users can now see that MoodRun is not simply choosing a generic workout plan, but using their mood input to shape the running experience.

Design Impact

This refinement directly addressed Finding 2 by making the adaptive logic visible inside the running plan interface. It increased users' trust in the recommendation, reduced confusion about why a plan was suggested, and strengthened MoodRun's human-centred design by showing how emotional input affects the run plan.

Final Reflection and Technical Reflection

Social and ethical implications of the design, plus AI use and verification.

Social and Ethical Reflection

MoodRun was designed as a mobile-first playful running companion for young people who may feel stressed, emotionally overloaded, or unmotivated to exercise. Its social value is that it reframes running as emotional self-care rather than only performance tracking. By starting with mood check-in, recommending a mood-based run plan, supporting route choice, and ending with reflection, MoodRun encourages users to notice their emotional state and connect physical movement with emotional change.

However, this design also involves ethical responsibilities. Mood selections, written reflections, GPS location, route choices, profile details, avatar choices, and run history can all be sensitive data. To reduce privacy risks, MoodRun does not use a custom backend. Prototype data is stored locally in the browser through localStorage, and GPS permission is only requested when route setup or live run tracking requires it. The system also uses AMap for map display and geocoding, so users should understand that map-related features depend on a third-party map service.

MoodRun's emotional features also need careful framing. The Mood AI assistant is not a clinical or diagnostic tool. It is a local rule-based mood classifier that suggests a possible mood from user text; it does not call an external generative AI API. Its output is presented as a supportive suggestion, not as a psychological judgement. Similarly, Wisdom Book reflections are motivational prompts, not professional mental health advice.

Accessibility and inclusion were also considered. MoodRun uses a playful retro visual identity, but the interface still needs to remain readable, responsive, and usable on mobile devices. The design avoids punishing users for not following a route perfectly: routes are treated as recommendations, and users can continue free running even when they move away from the suggested path. This supports a less judgemental and more human-centred running experience.

Use of AI in This Project

Generative AI was used for system development and documentation support, in line with the coursework permission for Portfolio, System, and Video Demo submissions. The core project motivation, design direction, user research, usability findings, and human-centred justification remained the group's own work. AI was used as an implementation assistant, not as a replacement for design decision-making.

The main AI-assisted prompts are documented in the /ai-logs folder. These included:

  • Refactoring the original MoodRun prototype into a clearer Vue 3 + TypeScript structure while preserving the retro visual identity, mood-to-run flow, localStorage compatibility, and documentation structure.
  • Adding AMap live map integration, including GPS path rendering, planned route overlays, current runner marker, GPS accuracy display, and map loading/error states.
  • Adding route setup before running, including map picking, manual search, random route generation, plan distance vs route distance comparison, and target mode selection.
  • Building the GPS metrics engine, including distance, pace, elapsed time, calories, remaining distance, progress, GPS quality labels, and filtering for inaccurate or unrealistic GPS points.
  • Adding route guidance during a run, including start-point warnings, off-route warnings, back-on-route feedback, and free-running behaviour.
  • Adding local Mood AI, mood-based run plans, mood profiles, checkpoint feedback, after-run mood outcomes, and Wisdom Book reflection.
  • Adding profile editing, avatar customisation, local run history, meditation audio, running music, voice coach, and optional voice commands.
  • Reviewing the system against coursework delivery requirements and preparing README / AI disclosure documentation.

Verification of AI-Assisted Code

AI-assisted code was not accepted blindly. We verified the implementation against MoodRun's user requirements and coursework requirements in several ways. First, we checked that the main user journey worked: home page, mood check-in, mood assistant, recommended plan, route setup, running mode, summary, Wisdom Book, and profile/history. Second, we ran production builds using npm run build after major implementation and documentation changes. Third, we reviewed specific risk areas, such as GPS permission denial, weak GPS status messages, unsupported speech features, map loading errors, audio cleanup when leaving pages, and localStorage compatibility for profile, avatar, and run history.

We also compared implementation decisions against the system's playful requirements. For example, route guidance was checked to ensure that users were warned but not punished when leaving the recommended route. The mood assistant and Wisdom Book were checked to ensure they used local data and rule-based logic rather than external generative AI calls.

Ethical Considerations of AI-Generated Code

Using AI for coding introduced several ethical considerations. First, AI-generated code can contain hidden assumptions, bugs, or accessibility issues, so we reviewed and tested generated features before including them in the final prototype. Second, because MoodRun handles mood text and location-related interaction, privacy risks had to be considered carefully. The final system avoids a custom backend and stores prototype data locally, but users should still be informed that localStorage is not the same as secure encrypted storage.

Third, rule-based mood classification may misunderstand user language, especially because emotional expression is personal and culturally specific. To reduce this risk, MoodRun presents the mood assistant's output as a suggestion rather than a final judgement. The system should not claim to diagnose mental health conditions. Finally, optional browser speech recognition may depend on browser or vendor services, so voice control should be treated as optional and clearly communicated to users.
